We are building a university Big Data Engineering project titled:

**Real-Time Crime Analytics and Intelligent Alert System**

The project uses Chicago public safety datasets and follows **Lambda Architecture** with three layers:

1. **Batch Layer** using Apache Spark/PySpark
2. **Speed Layer** using Kafka + Apache Storm
3. **Serving Layer** using PostgreSQL + MongoDB
4. Optional dashboard using Streamlit

We are using **Option B**, meaning Spark batch processing and Kafka + Storm real-time streaming will run **simultaneously**.

Spark processes historical datasets while Kafka + Storm process simulated live crime events. The dashboard must show real-time alerts immediately, even if Spark batch analytics are still running.

---

# 1. Core Architecture

```text
Historical CSV datasets
        ↓
Apache Spark batch jobs
        ↓
PostgreSQL analytical tables
        ↓
Streamlit dashboard historical views
```

At the same time:

```text
Crime CSV replayed row by row
        ↓
Kafka producer
        ↓
Kafka topic: crime-events
        ↓
Apache Storm topology
        ↓
ParseBolt → DistrictBolt → WindowBolt → AnomalyBolt → AlertBolt
        ↓
PostgreSQL alerts table + MongoDB alert_logs collection
        ↓
Streamlit dashboard real-time views
```

Spark and Storm must not duplicate each other’s work.

Spark answers:

```text
What has happened historically?
```

Kafka + Storm answer:

```text
What is happening right now?
```

The dashboard combines both.

---

# 2. Key System Rules

The system must follow these rules:

1. Spark and Storm run simultaneously.
2. Storm real-time alerts must work even if Spark has not finished.
3. Dashboard must not crash when historical tables are empty.
4. Spark must never expose partial batch results.
5. Spark and Storm must write to separate tables.
6. All shared field names must be standardized.
7. `district` must be treated as `STRING` everywhere.
8. All batch outputs must include a deterministic `run_id`.
9. Dashboard must read only the latest completed Spark run.
10. MongoDB raw event logging must be disabled by default or controlled using TTL.

# 5. Standard Data Contract

All systems must use the same cleaned field names.

Kafka JSON message format:

```json
{
  "case_number": "...",
  "date": "...",
  "block": "...",
  "primary_type": "...",
  "district": "...",
  "arrest": true,
  "latitude": 41.0,
  "longitude": -87.0
}
```

Important rule:

```text
district must always be STRING.
```

Even if the original CSV has district as integer or null, convert it:

```python
col("district").cast("string")
```

If district is null:

```text
district = "UNKNOWN"
```

Do not use mixed names such as:

```text
CASE NUMBER
caseNumber
case_no
CaseNumber
```

Use:

```text
case_number
district
primary_type
latitude
longitude
```

# 7. Run ID Rule

Every Spark batch run must generate a deterministic `run_id`:

```text
batch_YYYYMMDD_HHMMSS
```

Example:

```text
batch_20260501_173000
```

Generate it once at the start of `run_batch.py` and pass it to all Spark jobs.

Do not generate a separate `run_id` for each analytics script.

---

# 8. Latest Completed Run Query

Dashboard must use this query to find the latest valid batch output:

```sql
SELECT run_id
FROM batch_job_status
WHERE status = 'completed'
ORDER BY finished_at DESC
LIMIT 1;
```

Then dashboard must use that `run_id` when reading batch tables:

```sql
SELECT *
FROM crime_trends
WHERE run_id = '<latest_completed_run_id>';
```

If no completed run exists, dashboard must show:

```text
Historical analytics are still being generated. Live alerts are active.
```

---

# 9. Spark Write Strategy

Spark must not write partial results directly into final tables.

Use this pattern:

```text
1. Generate run_id.
2. Mark job as running in batch_job_status.
3. Write analytics results to temp/staging tables.
4. When all Spark jobs finish successfully:
   - publish staging results into final tables
   - mark batch_job_status as completed
5. If Spark fails:
   - mark batch_job_status as failed
   - dashboard continues using previous completed run
```

Example:

```text
crime_trends_temp
arrest_rates_temp
violence_stats_temp
sex_offender_density_temp
hotspots_temp
correlations_temp
```

Only after Spark succeeds should data move into:

```text
crime_trends
arrest_rates
violence_stats
sex_offender_density
hotspots
correlations
```

# 11. Publishing Staging Results

After Spark successfully writes all temp tables, publish results into final tables.

Example:

```sql
INSERT INTO crime_trends (
    run_id,
    trend_type,
    trend_key,
    crime_count,
    created_at
)
SELECT
    run_id,
    trend_type,
    trend_key,
    crime_count,
    created_at
FROM crime_trends_temp
WHERE run_id = '<current_run_id>';
```

Do this for all temp tables.

After successful publishing, mark run as completed:

```sql
UPDATE batch_job_status
SET status = 'completed',
    finished_at = CURRENT_TIMESTAMP,
    last_successful_run = CURRENT_TIMESTAMP,
    message = 'Batch analytics completed successfully'
WHERE run_id = '<current_run_id>';
```

If anything fails:

```sql
UPDATE batch_job_status
SET status = 'failed',
    finished_at = CURRENT_TIMESTAMP,
    message = '<error message>'
WHERE run_id = '<current_run_id>';
```

Dashboard must ignore failed and running runs.

# 13. Severity Logic

Storm must assign severity using this rule:

```text
If event_count >= high_min:
    severity = HIGH

Else if event_count >= medium_min:
    severity = MEDIUM

Else if event_count >= low_min:
    severity = LOW

Else:
    no alert
```

Using config:

```yaml
severity:
  low_min: 25
  medium_min: 40
  high_min: 60
```

Example:

```text
event_count = 27 → LOW
event_count = 45 → MEDIUM
event_count = 65 → HIGH
event_count = 20 → no alert
```

# 19. No Duplicate Processing Rule

Avoid redundant processing:

Spark must not:

```text
consume Kafka
generate real-time alerts
write to alerts table
write to realtime_district_counts table
```

Storm must not:

```text
run historical trend analysis
run K-Means clustering
join all historical datasets
write to Spark-owned analytics tables
```

Dashboard must not:

```text
perform major analytics
clean raw datasets
calculate K-Means
replace Spark or Storm logic
```

Dashboard only reads and displays results.
